# HW1 - Architectural Kata: Ticket to Serve

## Полученное условие

**Ticket to Serve**

Централизованный ticketing API для продавцов билетов (реселлеров). Реселлеры интегрируют API в свои системы.

**Пользователи:** несколько тысяч реселлеров; у каждого - от тысяч до миллионов конечных пользователей.

**Требования:**
- API: список доступных билетов на шоу + покупка
- Уведомление пользователя, когда интересующий билет только что купили
- **Гарантия:** один билет не может быть куплен дважды
- Admin-интерфейс для поддержки реселлеров
- Комиссионная система (fee) с каждой покупки
- Event-интерфейс для реселлеров (например, фрод-платёж через несколько дней)
- Бухгалтерия: прозрачные отчёты для реселлеров

**Бонус:** хостинг для реселлеров; разделение клиентской базы по законодательству разных стран.

## Решение

## 1. Отсев требований

### Ключевые вопросы стейкхолдерам

| Вопрос | Ответ |
|--------|-------|
| Что будет, если не сделать централизованный API? | Каждый реселлер дублирует логику бронирования, из чего следуют двойные продажи, расхождения в учёте, рост затрат на интеграции |
| Как измерить успех? | 0 двойных продаж; latency API < 300 ms p95; 99.9% uptime; время онбординга реселлера < 2 недели |
| Реалистичность в РФ 2026? | Зарубежные облака и платёжные системы недоступны; нужны отечественные ЦОД, СБП/МИР, соответствие ФЗ-152 и требованиям ИБ |

### Итог отсева

| Включено (MVP) | Исключено / отложено |
|----------------|----------------------|
| API каталога + покупки | Хостинг реселлеров |
| Идемпотентная покупка, антидубль | Полное разделение по странам |
| Webhook-события для реселлеров | Сложная ML-антифрод система |
| Admin-панель L1-поддержки | Мультивалютность beyond RUB |
| Fee + базовая бухгалтерия | |

## 2. Scope Refinement

### Пользователи

| Роль | Кол-во | Действия |
|------|--------|----------|
| **Покупатель** | миллионы (через реселлеров) | Просмотр билетов, покупка, подписка на уведомления |
| **Реселлер** | ~3000 | Интеграция API, настройка комиссий, отчёты |
| **Оператор поддержки** | ~50 | Admin: ручные операции, разбор инцидентов |
| **Администратор платформы** | ~10 | Управление шоу, тарифами, доступами |

### Основные функции (MVP)

**Для покупателя (через API реселлера):**
- `GET /shows/{id}/tickets` - доступные места
- `POST /orders` - резерв + оплата (идемпотентный `Idempotency-Key`)
- `POST /subscriptions` - уведомить, когда место освободилось/появилось

**Для реселлера:**
- API-ключи, webhook endpoint
- Дашборд: продажи, комиссии, возвраты
- Sandbox для тестирования интеграции

**Для поддержки:**
- Поиск заказа, принудительный возврат, блокировка билета
- Просмотр audit log

### Дополнительные функции (фаза 2)

- White-label хостинг витрины реселлера
- Geo-sharding по юрисдикциям (отдельный кластер для ЕС при расширении)
- Расширенный антифрод (scoring, delayed capture)

### Архитектурные компоненты

| Модуль | Функционал |
|--------|------------|
| **Ticket API Gateway** | Auth, rate limit, routing, OpenAPI 3.0 |
| **Inventory Service** | Состояние мест, pessimistic/optimistic lock |
| **Order Service** | Оформление, идемпотентность, saga оплаты |
| **Notification Service** | WebSocket/SSE + webhooks реселлерам |
| **Billing Service** | Fee, взаиморасчёты, акты |
| **Admin Portal** | Операции поддержки |
| **Event Bus** | Kafka / отечественный аналог (Platform V, Yandex Data Streams) |

## 3. Функциональные требования

### Акторы

- **Покупатель**, **Реселлер**, **Оператор**, **Администратор**, **Платёжный провайдер** (СБП, эквайринг)

### Домен и данные

| Сущность | Ключевые поля |
|----------|---------------|
| Show | id, venue, datetime, status |
| Seat / Ticket | id, show_id, sector, row, seat, price, status (available/reserved/sold) |
| Order | id, reseller_id, buyer_ref, tickets[], amount, status |
| Payment | id, order_id, provider, status, fraud_flag |
| Fee | order_id, platform_fee, reseller_payout |

### Бизнес-процессы

1. **Покупка:** резерв места (TTL 10 мин) → оплата → подтверждение → webhook реселлеру
2. **Антидубль:** `SELECT ... FOR UPDATE` или distributed lock (Redis) на `seat_id`
3. **Уведомление:** при смене статуса места push в очередь и webhook/SSE
4. **Фрод:** delayed event через N дней → webhook `payment.flagged`

### User Stories

1. *Как покупатель*, хочу получить уведомление, когда билет на нужное место освободился - критерий: webhook/SSE < 5 с после освобождения.
2. *Как реселлер*, хочу гарантию, что один seat_id не продан дважды - критерий: 0 инцидентов двойной продажи, тесты гонок.
3. *Как оператор*, хочу отменить заказ и вернуть место в продажу - критерий: операция в audit log, webhook реселлеру.

### Корнер-кейсы

- Двойной клик «Купить» это идемпотентность по `Idempotency-Key`
- Оплата прошла, но webhook не доставился → retry + DLQ + ручной reconcile
- Реселлер недоступен, значит circuit breaker, события в DLQ до 72 ч

## 4. Нефункциональные требования

### Производительность

| Метрика | Значение |
|---------|----------|
| Пиковый RPS API | 15 000 (см. расчёт) |
| Latency | < 300 ms p95 для каталога; < 1 с для покупки |
| Двойная продажа | 0 допустимо |

### Масштабируемость

- Горизонтальное масштабирование stateless-сервисов
- Шардирование inventory по `show_id`
- Event-driven для уведомлений

### Отказоустойчивость

- SLA 99.9%
- RTO < 15 мин, RPO < 1 мин (синхронная реплика БД)
- Multi-AZ в российском облаке (Yandex Cloud / VK Cloud / SberCloud)

### Безопасность и ИБ (РФ 2026)

| Требование | Реализация |
|------------|------------|
| ФЗ-152 | ПДн только в ЦОД на территории РФ |
| СТО БР ИББС / ГОСТ Р 57580 | Сегментация сети, RBAC, журналирование |
| Сертификация ИБ | План ФСТЭК/ФСБ на уровне ИСПДн; шифрование TLS 1.3, AES-256 at rest |
| PCI DSS (платежи) | Токенизация карт, PCI через сертифицированного провайдера |
| API security | OAuth 2.0 / mTLS для реселлеров, rate limiting |

### Мониторинг

- Prometheus + Grafana (или отечественный MONITORING)
- Алерты: двойная продажа (бизнес-метрика!), рост 5xx, lag Kafka
- Audit log всех операций поддержки (неизменяемый)

## 6. Расчёт нагрузки

### Исходные допущения

| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| Реселлеров | 3 000 | из условия |
| DAU покупателей (суммарно) | 500 000 | ~170 покупателей/реселлер в день (средний) |
| Запросов за сессию | 12 | каталог, выбор места, 2x polling, оплата, подтверждение |
| Сессий/день на покупателя | 1.5 | часть уходит без покупки |
| Покупок/день | 50 000 | ~10% DAU конвертируется |

### RPS по формуле

```
RPD = DAU x запросов_за_сессию x сессий_на_DAU
RPS_avg = RPD / 86 400
RPS_peak = RPD x доля_пикового_трафика / длина_пика_сек
```

### Каталог и просмотр

| Метрика | Расчёт | Значение |
|---------|--------|----------|
| RPD | 500 000 x 12 x 1.5 | 9 000 000 |
| RPS средний | 9M / 86 400 | **~104 RPS** |
| Вечерний пик (19–23 MSK, 40% трафика) | 9M x 0.4 / 14 400 | **~250 RPS** |
| Пик продажи топ-шоу (x10) | | **~2 500 RPS** |

### Покупки (write-path)

| Метрика | Расчёт | Значение |
|---------|--------|----------|
| TPS средний | 50 000 / 86 400 | **~0.6 TPS** |
| Пик (5% за 1 час) | 50 000 x 0.05 / 3 600 | **~0.7 TPS** |
| Мега-концерт (10% дневных продаж за 15 мин) | 5 000 / 900 | **~5.5 TPS** |

> Write-нагрузка невелика, но **критична по корректности** (lock, транзакции).

### Уведомления / webhooks

| Метрика | Значение |
|---------|----------|
| Событий/день | ~200 000 (просмотры + покупки + отмены) |
| Пик SSE/webhook | **~500 events/s** при мега-концерте |

### Одновременные пользователи

| Метрика | Значение |
|---------|----------|
| Средние concurrent | 500 000 x (10 мин / 1 440 мин) ≈ **3 500** |
| Пик (мега-концерт) | **30 000–50 000** |

### Сеть и хранение

| Тип | Объём |
|-----|-------|
| API-трафик (JSON) | ~50 ГБ/мес |
| Audit / event log | ~100 ГБ/мес |
| Метаданные билетов (10 KB × 50K покупок x 365) | ~180 ГБ/год |

### Инфраструктура (оценка)

| Компонент | vCPU | RAM | Нод | Обоснование |
|-----------|------|-----|-----|-------------|
| API Gateway | 4 | 8 GB | 3 | ~2 500 RPS пик |
| Inventory + Order | 8 | 16 GB | 4 | lock-heavy writes |
| PostgreSQL | 16 | 64 GB | 3 (1 primary + 2 replica) | транзакции, HA |
| Kafka (3 broker) | 4 | 16 GB | 3 | events, webhooks |
| Redis (lock/cache) | 4 | 16 GB | 3 | seat locks, idempotency |

**Итого:** ~80 vCPU, 320 GB RAM, 1 Gbit/s канал.

## Итог

Система **Ticket to Serve** в контексте РФ 2026 - это:

1. **Read-heavy API** (каталог) + **write-critical path** (покупка с гарантией уникальности).
2. Ключевой риск - не RPS, а **корректность инвентаря** и **соответствие ИБ** (ФЗ-152, сертификация).
3. MVP фокусируется на API + admin + billing; хостинг и geo-sharding - фаза 2.
4. Расчёт показывает ~250 RPS вечерний пик, до **2 500 RPS** при топ-шоу - нужен горизонтальный масштаб API и кэш каталога.
